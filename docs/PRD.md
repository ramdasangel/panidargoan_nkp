## 1. Executive summary

PaniDarGoan is a watershed-management and water-source-monitoring platform for rural India. It pairs an authoritative geographic skeleton (state → district → taluka → village; river basin → micro-watershed) with the ability for **field users to capture, edit, and continuously monitor individual water sources** (wells, check dams, rivers, ponds, etc.) over time, attaching photos and PDFs as evidence.

The pilot covers two talukas (Ambegaon, Shirur) in Pune district. The data model and deployment are designed to scale to the rest of Maharashtra and to all of India without re-architecting.

## 2. Background & motivation

Maharashtra watershed agencies (and similar bodies in other Indian states) currently track interventions and water-source health through a patchwork of spreadsheets, paper field books and PDF reports. There is no single living spatial record that ties **monitoring observations** (flow, water level, pH, TDS, turbidity), **investments** (projects, tasks, cost entries), and **physical structures** (wells, bandharas, check dams, percolation tanks) to a **shared geography**.

PaniDarGoan provides that single record.

## 3. Goals and non-goals

### Goals (in scope for v0.x)
- A foldable left-hand menu with toggleable map layers and an admin section.
- Real administrative boundaries (state → village) imported from OSM, with village polygons generated by Voronoi tessellation where OSM polygons aren't available.
- A watershed hierarchy (Bhima → Ghod → Ambegaon micro / Shirur micro) overlaid on the same map.
- Water sources with two origins:
  1. **Auto-imported** from OpenStreetMap (rivers, streams, canals) — read-only.
  2. **Manually mapped** by field users — full edit + master/detail logging.
- For each manual water source: a time-stamped **log entry** with CGWB / CWC / IS 10500–style attributes (flow m³/day, water level cm, pH, TDS ppm, turbidity NTU, temperature °C, condition, notes).
- File attachments per log entry (photos, PDFs); only the URL persists in PostgreSQL — files live in MinIO.
- Place search (Nominatim autocomplete) and "Center on my device" geolocation.
- Google Sign-In + role-based access (admin, project_manager, field_user, viewer).
- Admin can pre-provision users by email + role.

### Non-goals (deferred)
- Project / task / cost-entry CRUD UI (schema exists; UI was removed in v0.x to focus on watershed/monitoring).
- Full Indian states / districts beyond Maharashtra pilot.
- Offline data capture / sync queue.
- Native mobile apps. The PWA covers Android/iOS through the browser.

## 4. Target users and roles

| Role | Who | Capabilities |
|---|---|---|
| **admin** | Programme managers, NGO leads | Everything; can manage users + delete records |
| **project_manager** | Watershed engineers | View all; create + edit water sources; add log entries; delete attachments |
| **field_user** | Field staff with smartphones | View all; create water sources; add log entries; upload photos |
| **viewer** | Donors, dashboard consumers | Read-only |

Auth is **Google OAuth 2.0** (existing dummy-login mode kept for demos). New Google users are auto-created on first sign-in. Emails listed in `ADMIN_EMAILS` env get the admin role automatically; everyone else starts as `viewer` until an admin promotes them.

## 5. Functional requirements

### 5.1 Map view (default landing)
- **Foldable left sidebar** with three sections (Layers, Actions, Watersheds) plus an Admin section visible only to admins.
- **Place search** (Nominatim, debounced, India-only) at top-centre.
- **Center on device** button next to the search → flies to user's GPS at walking-scale zoom 17.
- **Base map:** OpenStreetMap default; toggle to **OpenTopoMap** for SRTM-derived hillshade + contour lines.
- **Layers (independently toggleable):**
  - Villages (real Voronoi-clipped boundaries)
  - Talukas (real OSM polygons)
  - Watersheds (Bhima → Ghod → micro)
  - Water sources (manually mapped)
  - Water sources (auto sourced from OSM)
- **Watershed tree** in the sidebar showing the hierarchy with click-to-fly.

### 5.2 Add water source flow
- "Add water source" button under the Actions section of the sidebar.
- Type selector → 14 types grouped by geometry (Point / Line / Polygon).
- Two input modes:
  1. **Draw on map** — click to drop vertices; the panel can be collapsed via a labelled "Map" button so the map is fully visible while drawing. A pulsing collapsed bar with a "Form" button brings the form back.
  2. **Paste / upload KML** — Nominatim-style geometry import.
- Vertex count, length (line) or area (polygon) shown live.
- On save, server also generates a KML snapshot from the geometry if none was supplied; both PostGIS geometry and KML text persist.

### 5.3 Water source master/detail (manual sources only)
- Pulsing orange centroid marker per manually-mapped source.
- Click → side panel:
  - **Master:** inline view + edit (name, capacity, depth, condition, notes).
  - **Detail (logs):** chronological log entries with stat pills and warning highlight when potable thresholds are exceeded (pH outside 6.5–8.5, TDS > 500 ppm, turbidity > 5 NTU).
  - "Add log" form with CGWB/CWC/IS 10500 attributes + file picker (image/* and PDF, up to 10 files × 10 MB).

### 5.4 Admin: user management
- Sidebar → Admin → Manage users (admin role only).
- Modal with users grouped by role; add by email + role; inline role change; safe deletion (cannot delete self; cannot delete last admin).

## 7. Architecture

```
                Browser (React PWA)
                       │
                       │ HTTPS
                       ▼
              ┌─────────────────┐
              │     Caddy       │  port 80/443, Let's Encrypt
              │ (reverse proxy) │
              └────────┬────────┘
                       │
              ┌────────▼────────┐
              │ web (nginx)     │  static SPA + /api proxy
              └────────┬────────┘
                       │
              ┌────────▼────────┐
              │  api (Express)  │  Prisma · S3 SDK · multer
              └──┬───┬───┬──────┘
                 │   │   │
       ┌─────────┘   │   └─────────┐
       │             │             │
┌──────▼────┐ ┌──────▼────┐ ┌──────▼────┐
│ Postgres  │ │   Redis   │ │   MinIO   │
│ + PostGIS │ │  (cache)  │ │ (objects) │
└───────────┘ └───────────┘ └───────────┘
```

All six containers run on a **single Hostinger VPS** under a single `docker compose` project, with internal-only networking for the data tier. The only ports exposed publicly are 80/443 (Caddy) and 22 (SSH). Local-dev runs the same compose without the `https` profile.

## 8. Data model (key tables)

| Table | Purpose |
|---|---|
| User | Email-keyed; Google `sub` linked on first sign-in; role enum |
| Country / State / District / Taluka / Village | Real OSM polygons + Voronoi villages |
| Watershed | Self-referencing tree (level + kind + boundary) |
| WaterSource | `geom geography(Geometry,4326)` + `kml text`; `source` enum (manual / osm / imported) |
| WaterSourceLog | Time-stamped readings with CGWB-style attributes |
| WaterSourceLogAttachment | `url` only (file lives in MinIO) |
| Project / Task / CostEntry / TaskGeoLink / Resource | Schema present; UI deferred |

Spatial indexing (GIST) on every boundary column. Audit columns (`createdAt`, `loggedAt`) where useful.

## 9. Deployment architecture

| Layer | Image | Volume | Public port |
|---|---|---|---|
| db | `panidargoan-db:local` (PostGIS 16-3.4 + custom init) | `pgdata` | none in prod |
| redis | `redis:7-alpine` | `redisdata` | none |
| minio | `minio/minio:latest` | `minio_data` | 9001 (console) only |
| api | `node:20-bookworm-slim` | — | none |
| web | `nginx:1.27-alpine` | — | none |
| caddy | `caddy:2-alpine` | `caddy_data`, `caddy_config` | 80, 443 |

- Single source of truth: `docker-compose.yml`.
- Compose **profiles** separate dev (db + redis + minio) from full (`app`) from production HTTPS (`https`).
- Code on the VPS is just a `git clone` of `main`. Deploy = `git pull && docker compose --profile app --profile https up -d --build`.

## 11. Security

- **Transport:** HTTPS only on production (Caddy with Let's Encrypt + auto-renew). HTTP redirects 308 → HTTPS.
- **Auth:** Google ID-token verified server-side via `google-auth-library`. Our own JWT with 7-day TTL for sessions. Role checked on every protected route.
- **Secrets:** `.env` files gitignored; `client_secret_*.json` gitignored. Random `JWT_SECRET` and `MINIO_ROOT_PASSWORD` generated at first deploy.
- **CORS:** Server allows the configured `CORS_ORIGIN` only.
- **DB:** PostGIS public-schema ownership transferred to the app user via `db/init.sql` on first volume init.
- **MinIO:** No public port; the API streams objects, so file URLs cannot be hot-linked without going through auth-aware logging at the API layer (future: enforce auth on `/api/uploads/*` if needed).
- **Attachments:** Mime whitelist (jpeg/png/webp/gif/heic/pdf), 10 MB size cap, 10 files per upload.

## 12. Operations

- **Backups (recommended):**
  - `pg_dump` weekly to a separate machine or S3/B2 bucket.
  - MinIO bucket replication / weekly tarball.
- **Monitoring (recommended for production):**
  - UptimeRobot or BetterStack free tier hitting `/api/health` every 5 min.
  - Hostinger panel for raw VPS CPU / RAM / disk graphs.
- **Logging:** `docker compose logs <service> --tail=N`. No centralised log aggregator yet.

## 14. Risks and open questions

1. **Indian village boundary data quality** — Voronoi cells are a credible approximation but not gazetteer-correct. For production we'll need ground-truthed Census 2011 shapefiles.
2. **Nominatim rate limiting** — 1 req/sec policy is enough for interactive search but not for bulk operations. If we hit limits we self-host Nominatim (adds ~2 GB RAM + ~30 GB disk on the VPS).
3. **OAuth lock-in** — currently single provider (Google). Add Apple Sign-In if pivoting to iOS users.
4. **MinIO disk growth** — needs a backup target before we have meaningful field-photo volume.
5. **Single point of failure** — one VPS. SLO is "best effort". Acceptable for a pilot but plan a managed-DB move before stakeholder reporting becomes mission-critical.

## Appendix A — Glossary

- **CGWB** — Central Ground Water Board, India's authority on groundwater data.
- **CWC** — Central Water Commission, surface-water authority.
- **IS 10500** — Bureau of Indian Standards drinking-water spec; defines pH/TDS/turbidity acceptability.
- **Voronoi tessellation** — Algorithm that partitions a plane into cells, one per seed point, where each cell contains all points closer to its seed than to any other. Used to approximate village boundaries from village-centre coordinates.
- **PWA** — Progressive Web App. Installable from the browser, behaves like a native app.
